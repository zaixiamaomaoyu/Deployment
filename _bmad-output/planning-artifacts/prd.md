# Product Requirements Document - Deployment

**Author:** maomaoyu
**Date:** 2026-03-24

## Executive Summary

**前端部署学习平台**是一个面向前端开发者的交互式学习产品，旨在帮助零部署经验的开发者从「第 0 步」开始掌握前端包部署技能。产品采用 Web 前端 + AI 助手后端的架构，为用户提供引导式学习体验。

**目标用户：**
- **入门开发者**：有一定前端开发能力，但从未部署过项目，不知道需要准备什么
- **进阶开发者**：有部署经验但存在知识盲区，希望系统化提升

**核心问题：** 现有部署教程假设用户已具备基础设施（服务器、账号、环境），忽略了新手真正的卡点——「我不知道我需要什么，也不知道从哪里开始」。用户在教程中途遇到问题时缺乏即时帮助，导致学习中断。

**产品形态：**
- 结构化知识体系：6 大领域（构建、平台、服务器、自动化、域名、容器）× 5 个层级 = 30 个知识点
- AI 学习教练：概念解释、术语解答、简单问题诊断（MVP 阶段）
- 游戏化激励：成就徽章、技能雷达图、学习进度追踪（Vision 阶段）

### What Makes This Special

| 差异化维度 | 本产品 | 传统方案 |
|-----------|--------|---------|
| **起点** | 从「不知道要准备什么」开始 | 假设已有环境/账号 |
| **AI 角色** | 引导式学习教练，主动推进 | 被动问答或无 AI |
| **学习体验** | 陪伴式、即时反馈、游戏化 | 孤立阅读、遇到问题即卡住 |

**核心洞察：** 填补「第 0 步」空白——在用户动手之前，先帮他们理清「我需要准备什么、我该选哪个方案」。

**成功标志：** 用户说「这个网站太棒了，我第一次觉得部署没那么难，终于有人从我的起点开始教了」并推荐给朋友。

## User Journeys

### Journey 1: 小林的第一次部署

**角色：小林，入门开发者**

**开场场景**

小林是一个刚工作半年的前端开发。她用 Vue 3 做了一个精美的个人作品集网站，`npm run build` 后生成了 dist 文件夹。她打开浏览器搜索「Vue 项目怎么部署」，看到了一堆教程：有的说买服务器，有的说用 Vercel，有的说要配置 Nginx...

她点开第一个教程，看到「SSH 连接服务器」就懵了——服务器是什么？要买吗？多少钱？她感到无从下手。

**发展过程**

小林打开前端部署学习平台。首页问她：「你准备好部署了吗？让我们先看看你需要什么。」

系统通过决策树引导她：
- 「你有服务器吗？」→ 没有
- 「你想花钱吗？」→ 不想
- 「你的项目是静态网站还是需要后端？」→ 静态网站
- 系统推荐：「Vercel 或 Netlify 适合你，免费且零配置」

小林跟着平台的步骤指南：
1. 注册 Vercel 账号
2. 连接 GitHub 仓库
3. 一键部署

每一步都有截图和说明，遇到不懂的术语，她点击「AI 助手」问：「什么是 GitHub 仓库？」AI 助手用简单的语言解释给她听。

**高潮时刻**

5 分钟后，小林看到了自己网站的线上地址。她点开链接，看到了自己的作品集在互联网上运行。

她兴奋地给朋友发了链接：「看，这是我做的网站！」

**结局**

小林第一次独立完成了部署。她意识到部署没有想象中那么难，只是之前没人从她的起点教她。她在平台上看到还有「服务器部署」的知识点，收藏了下来，打算以后学习。

### Journey 2: 张工的系统化学习

**角色：张工，进阶开发者**

**开场场景**

张工有 3 年前端经验，用过 Vercel 部署过几个项目。但他心里清楚，自己对部署的理解很浅——遇到报错就搜索，复制粘贴命令，解决了也不知道为什么。

他想深入学习服务器部署，但网上教程太散：
- 这篇讲 Nginx 配置，那篇讲 HTTPS 证书
- 有的教程用的 CentOS，有的用的 Ubuntu
- 命令敲下去报错，不知道是环境问题还是操作问题

他觉得需要有人帮他系统梳理一下。

**发展过程**

张工打开前端部署学习平台。他看到「技能雷达图」功能，评估了自己的部署知识水平：
- 构建：Lv3（熟悉）
- 平台：Lv2（了解）
- 服务器：Lv1（入门）
- 自动化：Lv1（入门）
- 域名：Lv0（未接触）
- 容器：Lv0（未接触）

系统推荐他学习「服务器」和「域名」领域的知识。

张工点开「服务器 Lv2：购买与 SSH 连接」。内容清晰告诉他：
- 服务器怎么选（配置、地域、价格对比）
- 免费试用方案有哪些坑
- SSH 连接的完整步骤

他跟着教程买了一台入门级服务器，成功 SSH 连接。遇到一个权限问题，问了 AI 助手，AI 帮他定位到是文件权限设置问题，并解释了原因。

**高潮时刻**

张工按照教程配置好 Nginx，把自己的项目部署到了自己的服务器上。他看着终端里的「Welcome to nginx!」页面，第一次觉得自己真正理解了部署。

**结局**

张工的学习进度被平台记录下来。他计划下个月学习「自动化」领域，把 GitHub Actions 搞明白。他在同事群里分享了网站链接：「这个平台不错，比到处找教程省心多了。」

### Journey 3: 管理员的内容运营

**角色：管理员**

**开场场景**

管理员收到一条用户反馈：「服务器那篇教程里的命令过时了，执行会报错。」她需要尽快更新内容。

**发展过程**

管理员登录后台管理系统。她在内容管理面板找到「服务器」领域的文章，定位到用户反馈的位置。

她检查了最新的文档，修正了命令，并添加了一条说明：「注意：CentOS 7 和 CentOS 8 的命令有差异，请根据你的系统版本选择。」

更新后，她在数据面板看到：
- 今日活跃用户：23 人
- AI 助手使用次数：47 次
- 用户最常搜索的问题：「Nginx 配置报错」

**高潮时刻**

管理员发现「Nginx 配置」是高频问题，决定新增一篇常见报错排查指南。她编写内容、提交审核、发布上线。

**结局**

管理员看到新文章上线后，相关问题的 AI 询问量下降了 30%。她感到平台正在帮助用户解决问题，这正是她想做的事情。

## Web App + API Backend Specific Requirements

### Project-Type Overview

| 维度 | 内容 |
|-----|------|
| 前端架构 | SPA 单页应用（Vue 3 + Element Plus） |
| 后端架构 | RESTful API + 实时通信 |
| 浏览器支持 | 现代浏览器（Chrome、Edge、Firefox、Safari 最新两个版本） |

### Technical Architecture Considerations

#### 前端架构

| 项目 | 决策 |
|-----|------|
| 框架 | Vue 3 |
| UI 组件库 | Element Plus |
| 路由 | Vue Router（SPA 路由） |
| 状态管理 | Pinia |
| 构建工具 | Vite |
| 实时通信 | WebSocket / SSE（用于 AI 流式输出） |

#### 后端架构

| 项目 | 决策 |
|-----|------|
| 认证方式 | 微信扫码登录 + Session |
| API 风格 | RESTful |
| 实时功能 | WebSocket / SSE（AI 流式输出） |
| 速率限制 | 无 |

### API Specification

| API 模块 | 端点 | 说明 |
|---------|------|------|
| 用户认证 | `GET /api/auth/wechat` | 微信扫码登录 |
| 用户认证 | `GET /api/auth/status` | 检查登录状态 |
| 用户认证 | `POST /api/auth/logout` | 退出登录 |
| AI 对话 | `POST /api/ai/chat` | AI 对话（支持流式响应） |
| 学习进度 | `GET /api/progress` | 获取用户学习进度 |
| 学习进度 | `POST /api/progress` | 保存学习进度 |
| 内容管理 | `GET /api/admin/content` | 获取内容列表 |
| 内容管理 | `POST /api/admin/content` | 新增内容 |
| 内容管理 | `PUT /api/admin/content/:id` | 更新内容 |
| 内容管理 | `DELETE /api/admin/content/:id` | 删除内容 |
| 数据统计 | `GET /api/admin/stats` | 获取统计数据 |

### Implementation Considerations

| 关注点 | 说明 |
|-------|------|
| AI 流式输出 | 使用 SSE（Server-Sent Events）或 WebSocket 实现实时响应 |
| Session 管理 | 服务端存储 Session，前端通过 Cookie 携带 |
| 微信登录集成 | 需要申请微信公众号/开放平台，获取 AppID 和 AppSecret |
| 管理员权限 | 需要实现角色区分（普通用户 vs 管理员） |

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

| 维度 | 内容 |
|-----|------|
| **MVP 类型** | 问题解决型 MVP |
| **核心验证目标** | AI 助手能否有效解决用户部署问题 |
| **最小团队** | 1 前端 + 1 后端 + 内容编写（可复用） |

**MVP 成功标准：** 用户能够通过决策树找到合适的部署方案，并通过 AI 助手解决遇到的问题，最终完成第一次部署。

### MVP Feature Set (Phase 1)

**Core User Journeys Supported:**
- 小林的第一次部署（决策树 + AI 问答 + 步骤指南）
- 管理员的基础内容管理

**Must-Have Capabilities:**

| 功能模块 | 具体内容 |
|---------|---------|
| 知识内容 | 6 大领域基础内容（Lv1-Lv2） |
| 决策树 | 部署方案引导选择 |
| 避坑指南 | 免费平台常见问题 |
| AI 助手（基础版） | 概念解释、术语解答、简单问题诊断 |
| 管理员后台 | 内容 CRUD、用户反馈查看 |
| 用户系统 | 微信扫码登录、学习进度保存 |

**MVP 不包含：**
- 技能雷达图
- 成就徽章
- AI 故障深度诊断
- 部署流程动画

**Phase 2 (Growth):**

|---------|---------|

| 技能雷达图 | 用户知识水平可视化评估 |
| 学习进度追踪 | 详细的学习记录和统计 |
| 个性化推荐 | 基于技能水平的学习路径推荐 |
| 数据统计 | 用户行为分析、热门问题统计 |

**Phase 3 (Vision):**

| 功能模块 | 具体内容 |
|---------|---------|
| 成就徽章系统 | 游戏化激励 |

| AI 教练陪伴 | 主动引导学习进度 |
| 社区功能 | 用户分享、问答（可选） |

### Risk Mitigation Strategy

| 风险类型 | 风险描述 | 缓解策略 |
|---------|---------|---------|
| **技术风险** | AI 助手回答质量不稳定 | MVP 先用高质量预设问答库，逐步接入大模型 |
| **市场风险** | 用户是否愿意用 AI 学习 | MVP 快速验证，收集用户反馈迭代 |
| **资源风险** | 开发资源不足 | MVP 精简功能，管理员后台可先用现成框架 |
